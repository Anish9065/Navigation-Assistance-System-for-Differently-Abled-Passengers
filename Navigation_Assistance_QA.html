Project Overview & Architecture

1 What is the Navigation Assistance System? Theory

The Navigation Assistance System is an AI-powered application designed to help differently-abled passengers navigate railway stations. It uses YOLOv8 object detection to detect passengers via camera feeds and provides real-time audio and visual guidance to help them reach their destination within the station.

The system identifies 6 classes of passengers: person, wheelchair user, blind person, person with crutches, elderly person, and person with luggage. It maps detected positions to predefined station zones and uses BFS (Breadth-First Search) to compute the shortest route to the destination.

7 Explain the overall data flow of the system from camera input to audio output. Theory
  1. Camera Capture: OpenCV captures video frames from webcam or video file.
  2. Object Detection: Each frame is passed to YOLOv8 model for inference → returns bounding boxes, class IDs, confidence scores.
  3. Result Formatting: Raw YOLO results are converted to structured dictionaries with class name, confidence, bbox coordinates, and center point.
  4. Zone Mapping: The center (cx, cy) of each detection is normalized to (fx, fy) and matched against predefined zone boundaries.
  5. Route Computation: BFS algorithm computes shortest path from detected zone to destination zone.
  6. Visualization: Annotated frame is drawn with bounding boxes, zone overlays, navigation arrows, and HUD.
  7. Audio Guide: The first step instruction is spoken via pyttsx3 (offline) or gTTS (online fallback) with debounce to avoid repetition.
  8. Output: Annotated frame displayed via OpenCV window or streamed as MJPEG to browser.

YOLOv8 & Object Detection

1 What is YOLO and what does it stand for? Theory

YOLO stands for "You Only Look Once". It is a real-time object detection algorithm that treats detection as a single regression problem, predicting bounding boxes and class probabilities directly from full images in one forward pass.

Unlike two-stage detectors (R-CNN), YOLO processes the entire image at once, making it significantly faster — enabling real-time detection at 30+ FPS.

2 What is YOLOv8 and why is it used in this project? Theory

YOLOv8 is the latest version by Ultralytics (2023). It is used because:

  • State-of-the-art accuracy — Best mAP among single-stage detectors
  • Easy API — Simple Python interface (from ultralytics import YOLO)
  • Multiple sizes — n (nano), s (small), m (medium), l (large), x (extra-large)
  • Anchor-free — Eliminates anchor box tuning
  • Built-in training — model.train() handles everything
  • COCO pretrained — Works out-of-the-box for person detection

4 What is confidence threshold and IoU threshold? Theory

Confidence Threshold (conf = 0.40): Minimum probability score a detection must have to be considered valid. Detections below this score are discarded. A higher threshold reduces false positives but may miss real objects.

IoU Threshold (iou = 0.45): Used in Non-Maximum Suppression (NMS). IoU = Intersection over Union. If two bounding boxes for the same class overlap with IoU > 0.45, the lower-confidence one is suppressed. This prevents duplicate detections.

Formula: IoU = Area of Overlap / Area of Union
5 What is Non-Maximum Suppression (NMS)? Theory

NMS is a post-processing step that removes redundant overlapping bounding boxes:

  1. Sort all detections by confidence score (descending)
  2. Select the highest-confidence box
  3. Remove all other boxes that have IoU > threshold with the selected box
  4. Repeat with remaining boxes

In this project, IoU threshold is set to 0.45 — if two boxes overlap more than 45%, the weaker one is removed.

6 What is the YOLO annotation format used in this project? Theory

YOLO format stores one .txt file per image. Each line represents one object:

<class_id> <center_x> <center_y> <width> <height>

All values are normalized (0 to 1) relative to image dimensions:

  • center_x = (x + w/2) / image_width
  • center_y = (y + h/2) / image_height
  • width = box_width / image_width
  • height = box_height / image_height

Example: 2 0.5123 0.4567 0.1250 0.3200 means class 2 (blind_person), centered at (51.2%, 45.7%), with width 12.5% and height 32% of the image.

9 How does the format_results() method work? Code
def format_results(self, results) -> list:
    if results is None: return []
    detections = []
    for box in results.boxes:
        cls_id = int(box.cls[0])               # class index
        name   = results.names.get(cls_id, ...) # class name
        conf   = float(box.conf[0])             # confidence
        x1, y1, x2, y2 = map(int, box.xyxy[0].tolist())  # bbox
        cx, cy = (x1+x2)//2, (y1+y2)//2        # center point
        detections.append({
            'class_id': cls_id, 'class_name': name,
            'confidence': round(conf, 3),
            'bbox': [x1,y1,x2,y2], 'center': [cx,cy]
        })
    return detections

It converts raw YOLO tensors into a clean Python list of dictionaries. The center point is used for zone mapping.

10 What is mAP (mean Average Precision) and how is it evaluated? Theory

mAP is the primary metric for evaluating object detection models:

  • Precision = True Positives / (True Positives + False Positives)
  • Recall = True Positives / (True Positives + False Negatives)
  • AP (Average Precision) = Area under the Precision-Recall curve for one class
  • mAP = Mean of AP across all classes

Two key metrics used in train.py:

  • mAP@0.5 — AP at IoU threshold 0.5
  • mAP@0.5:0.95 — Average of mAP at IoU thresholds 0.5, 0.55, ..., 0.95 (stricter)
metrics = model.val()
print(f"mAP@0.5    : {metrics.box.map50:.4f}")
print(f"mAP@0.5:0.95: {metrics.box.map:.4f}")

13 What are bounding boxes in object detection? Theory

A bounding box is a rectangle that encloses a detected object. It is defined by:

  • xyxy format: (x1, y1, x2, y2) — top-left and bottom-right corners (used in results.boxes.xyxy)
  • xywh format: (center_x, center_y, width, height) — used in YOLO label files

The detector extracts box.xyxy[0] for pixel coordinates and computes center as:

cx, cy = (x1 + x2) // 2, (y1 + y2) // 2
14 What is image size (imgsz) and why is 640 used? Theory

imgsz=640 means input images are resized to 640×640 pixels before feeding into the YOLO model. This is because:

  • Neural networks require fixed input dimensions
  • 640×640 is the default for YOLOv8 — optimized stride/grid alignment
  • Larger sizes (1280) improve small object detection but are slower
  • Smaller sizes (320) are faster but less accurate
15 What is batch size in training? Theory

Batch size is the number of images processed simultaneously in one training iteration. Default is batch=16.

  • Larger batch — More stable gradients, faster training (needs more GPU memory)
  • Smaller batch — Less memory, more noise in gradients, may generalize better
  • batch=-1 — Auto-batch: YOLO automatically determines the largest batch that fits in GPU memory
16 What is an epoch in training? Theory

An epoch is one complete pass through the entire training dataset. If you have 200 training images and batch size 16, one epoch = 200/16 = 13 iterations (batches).

The project defaults to 50 epochs. More epochs allow the model to learn better, but too many can cause overfitting (memorizing training data instead of generalizing).

20 What is anchor-free detection in YOLOv8? Theory

Earlier YOLO versions (v3, v5) used anchor boxes — predefined box shapes that the model adjusts. YOLOv8 is anchor-free, meaning:

  • It directly predicts the center point and dimensions of objects
  • No need to predefine anchor sizes/ratios
  • Simplifies training and reduces hyperparameter tuning
  • Better generalization to objects of unusual sizes

BFS Navigation & Graph Routing

1 What is BFS (Breadth-First Search)? Theory

BFS is a graph traversal algorithm that explores nodes level by level, starting from the source. It guarantees finding the shortest path in an unweighted graph.

  • Uses a queue (FIFO) data structure
  • Time complexity: O(V + E) where V = vertices, E = edges
  • Space complexity: O(V)
  • Explores all neighbors at current depth before moving deeper
2 Explain the BFS implementation in navigator.py. Code
def get_route(self, src, dst):
    if src == dst:
        return [src], ["You have arrived at your destination."]

    from collections import deque
    visited = {src: None}          # Track parent of each node
    queue   = deque([src])         # BFS queue

    while queue:
        node = queue.popleft()     # Dequeue front
        if node == dst:
            break                  # Found destination
        for (nxt, _, _) in STATION_GRAPH.get(node, []):
            if nxt not in visited:
                visited[nxt] = node  # Record parent
                queue.append(nxt)    # Enqueue neighbor

    # Reconstruct path by backtracking parents
    path = []
    cur = dst
    while cur is not None:
        path.append(cur)
        cur = visited.get(cur)
    path.reverse()

    # Build step-by-step instructions
    steps = []
    for i in range(len(path) - 1):
        for (nxt, instr, _) in STATION_GRAPH.get(path[i], []):
            if nxt == path[i+1]:
                steps.append(instr)
                break
    return path, steps

5 How is path reconstruction done after BFS? Theory

BFS stores the parent of each visited node in the visited dictionary. To reconstruct the path:

  1. Start at destination node
  2. Follow parent pointers backward: cur = visited.get(cur)
  3. Collect each node into a list
  4. Reverse the list to get source-to-destination order

Example: entrance → main_hall → platform_1

visited = {'entrance': None, 'main_hall': 'entrance', 'platform_1': 'main_hall'}
# Backtrack from platform_1: platform_1 → main_hall → entrance
# Reverse: [entrance, main_hall, platform_1]

9 Why is deque used instead of a regular list for BFS? Theory

collections.deque provides O(1) popleft() operations, while a regular list's pop(0) is O(n) because all elements must be shifted. For BFS:

append (enqueue)O(1)O(1)
pop(0) / popleft (dequeue)O(n)O(1)

This makes deque essential for efficient BFS in larger graphs.

Audio Guide & Text-to-Speech

1 What TTS engines does the AudioGuide use? Theory

The AudioGuide uses a fallback chain of 3 TTS methods:

  1. pyttsx3 (primary) — Offline TTS engine. Works without internet. Uses OS-native speech (SAPI5 on Windows, espeak on Linux, NSSpeech on macOS).
  2. gTTS (fallback) — Google Text-to-Speech. Requires internet. Generates MP3 files played via playsound.
  3. Console print (last resort) — If both TTS engines fail, prints the message to console.
2 Explain the speak() method in AudioGuide. Code
def speak(self, text: str):
    if not text: return
    with self._lock:                    # Thread-safe
        if self._engine:                # Try pyttsx3
            try:
                self._engine.say(text)
                self._engine.runAndWait()
                return
            except: pass
        try:                            # Fallback: gTTS
            from gtts import gTTS
            tts = gTTS(text=text, lang='en', slow=False)
            with tempfile.NamedTemporaryFile(delete=False, suffix='.mp3') as f:
                tmp = f.name
            tts.save(tmp)
            playsound.playsound(tmp)
            os.unlink(tmp)              # Delete temp file
        except:
            print(f"[AudioGuide] {text}")  # Last resort
3 What is debouncing and how is it used in speak_async()? Code
def speak_async(self, text: str, debounce: float = 4.0):
    now = time.time()
    if text == self._last_msg and (now - self._last_time) < debounce:
        return                        # Skip if same message within 4 seconds
    self._last_msg  = text
    self._last_time = now
    t = threading.Thread(target=self.speak, args=(text,), daemon=True)
    t.start()

Debouncing prevents the same audio message from being repeated too frequently. If the same instruction is triggered within 4 seconds, it is silently ignored. This prevents overlapping speech when detecting the same person across multiple frames.

4 Why is threading.Lock() used in the AudioGuide? Code

The self._lock ensures that only one thread can use the TTS engine at a time. Without it:

  • Multiple detection frames might trigger concurrent speech calls
  • pyttsx3 is not thread-safe — concurrent access crashes it
  • Two threads writing to the same audio output creates garbled sound

with self._lock: is a context manager that automatically acquires and releases the lock.

8 What is mosaic augmentation? Theory

Mosaic augmentation combines 4 training images into a single composite image by:

  1. Randomly selecting 4 images from the training set
  2. Cropping and resizing each to fit a quadrant
  3. Combining them into one image

Benefits: Exposes the model to more objects and context per batch, improves detection of small objects, and reduces the need for large batch sizes. Enabled with mosaic=1.0 (100% probability).
